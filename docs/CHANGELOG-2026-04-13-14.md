## 一、诚信工程 · Guard 系统（后端）

### 1.1 新增 integrity.go — 15 个 Backend Guard

文件：`server/internal/tools/integrity.go`（497 行新增）

每个 Guard 在 `save_output` 保存 `ontology_structure` 阶段时执行，拦截不合规的 YAML 并返回包含「什么错了 + 怎么改 + 示例」的中文 error message，引导 Agent 自愈。

| # | Guard | 拦截内容 | 来源 |
|---|-------|---------|------|
| 1 | `guardNoDuplicateMetricIDs` | 重复指标 ID | 对抗训练 R4 |
| 2 | `guardMetricRequiredFields` | kind/status/source_entities 缺失或值无效（如 kind=gauge/kpi, status=active） | S2 幻觉 |
| 3 | `guardTelemetryRequiredFields` | source_class/value_type/sampling/aggregations/status 缺失或错误字段名（如 source 代替 source_class, interval 代替 sampling） | S2 幻觉 |
| 4 | `guardImplementedMetricHasFormula` | status=implemented 但缺少 formula | 设计 |
| 5 | `guardClassificationHasBuckets` | kind=classification 但缺少 buckets 定义 | 对抗训练 R2 |
| 6 | `guardMetricSourceEntitiesExist` | source_entities 引用不存在的类 | 对抗训练 R2 |
| 7 | `guardNoPhantomDependencies` | depends_on 引用不存在的指标/遥测/类 + 自引用检测 | 对抗训练 R4 |
| 8 | `guardTelemetrySourceClassExists` | telemetry 的 source_class 引用不存在的类 | 对抗训练 R4 |
| 9 | `guardTelemetryHasContextStrategy` | context_strategy 缺失/为字符串/字段不完整 | 对抗训练 R5 |
| 10 | `guardNoFabricatedStandards` | 编造标准号（重复数字 GB/T 99999 / 未来年份 2030） | 对抗训练 R1 |
| 11 | `guardDerivedFormulaRefsExist` | 派生属性公式引用不存在的属性 | 对抗训练（技术总监人格） |
| 12 | `guardClassesInScene` | 跨阶段验证：类在场景分析中无对应描述（编造类检测） | 对抗训练（实习生人格） |
| 13 | `RunOntologyGuardsWithRaw` | 原始 YAML 全文扫描编造标准号（捕获不在 struct 字段中的额外属性） | 对抗训练 R1 |
| 14 | `autoFixRulesTrigger` | S3 规则 trigger 字符串自动修复为对象（"on_change" → {type: "on_change"}） | S3 幻觉 |
| 15 | `guardClassesInScene` | 读取 scene_analysis 阶段输出，对比验证所有类是否有来源 | H7 编造类检测 |

## 二、S2 任务切分（前端）

文件：`app/src/pages/AgentBuild/index.tsx`

### 改动
将 S2（本体架构师）从一次生成完整 YAML 改为三轮顺序调用：

| 轮次 | 任务 | 消息内容 |
|------|------|---------|
| Round 1 | 类 + 关系 | "只设计 classes 和 relationships，不要 metrics/telemetry" |
| Round 2 | 指标 | "read_ontology_structure → 追加 metrics（含 kind/status/source_entities 字段要求）" |
| Round 3 | 遥测 | "read_ontology_structure → 追加 telemetry（含 source_class/value_type/sampling 字段要求）" |

### 效果
- 每轮生成量从 ~8000 tokens 降到 ~2000 tokens
- Guard 在每轮 save 时独立验证
- 自愈成本从"重新生成整个 YAML"降为"只改 metrics/telemetry 部分"

## 三、Agent 配置修复

文件：`scripts/update-agents.sh`

| 修复 | 问题 | 影响 |
|------|------|------|
| S1 加 `list_documents` 到工具列表 | S1 无法获取文档列表，导致无法读取文档 | S1 读到空内容，输出通用模板 |

| S3 stage 名修正 | `rules_and_actions` → `rules_actions` | S3 保存到错误的 stage |
| 模型切换支持 | `deepseek-chat` ↔ `gpt-5.4` 可配置 | 支持多模型 |

## 五、Weave 平台改动

### 5.1 noDelegateDispatcher

文件：`internal/ontograph/graph.go`

- 新增 `noDelegateDispatcher` wrapper，从 LLM 可见的工具列表中过滤掉 `delegate`
- 如果 LLM 仍然尝试调 `delegate`，返回 error："delegate 工具已禁用。请直接使用你自己的工具完成任务。"
- 解决 DeepSeek 持续选择 delegate 而忽略 prompt 禁令的问题

### 5.2 auto_save 误报修复

文件：`internal/ontograph/graph.go`

- 新增 `checkStageHasOutput()` 函数：当 auto_save 在 LLM 输出中没找到 YAML 时，检查 DB 是否有已保存的 stage 输出
- 如果 Agent 已通过 tool call 自己保存（如 S1），显示"✅ 已保存"而非"⚠️ 未检测到结构化 YAML"

### 5.3 OpenAI Provider 支持

文件：`cmd/weave/main.go`

- 新增 `OPENAI_API_KEY` + `OPENAI_BASE_URL` 环境变量
- 注册 OpenAI provider，支持 gpt-5.4/gpt-5.4-mini/gpt-5.2 等模型
- `DEFAULT_MODEL` 环境变量可配置默认模型（默认 deepseek-chat）
- 注意：Loom 的 OpenAI client 在 baseURL 后追加 `/v1/chat/completions`，所以 baseURL 不要包含 `/v1`

### 5.4 预编译部署

文件：`deploy/Dockerfile.weave-prebuilt`、`deploy/deploy-weave.sh`、`deploy/docker-compose.yml`

| 文件 | 说明 |
|------|------|
| `Dockerfile.weave-prebuilt` | 6 行 Dockerfile，直接 COPY 预编译二进制，跳过 Docker 内 Go 编译 |
| `deploy-weave.sh` | 一键部署脚本：本地交叉编译 → 上传 → compose build → restart |
| `docker-compose.yml` | 加入 ontology-mcp、ontology-ui、neo4j 服务定义 + OpenAI 环境变量 |

解决的问题：
- Weave Dockerfile 依赖 `go get github.com/jinyitao123/loom@{commit}` 在 Docker 构建环境拉不到
- `docker compose up` 会用旧镜像覆盖手动 cp 的新二进制
- ontology-mcp/ontology-ui 不在 compose 里导致 DNS 别名不通
