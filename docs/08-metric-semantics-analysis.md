# 本体中的指标语义化问题

**从备件管理场景出发的分析**

v1.0 · 2026.03

---

> *本体告诉 Agent"世界长什么样"。但如果本体没有告诉 Agent"怎么衡量这个世界"，Agent 就只能猜。猜错了，管理者会失去信任。*

---

## 一、问题的起源

永祥科技的备件管理系统里有一个工具叫 `get_inventory_health`，它返回一个 0-100 的健康分。Agent 拿到这个分数后会告诉工段长："当前库存健康度 72 分。"

工段长问："72 分是怎么算的？"

Agent 答不上来。

不是因为 Agent 笨——是因为没有人告诉它这个分数怎么算的。健康分的计算公式埋在 `health.go` 的第 44-49 行：

```go
distancePenalty := math.Min(30, math.Max(0, (tv-targetValue)/targetValue*30))
stalePenalty := math.Min(30, staleRatio*100)
riskPenalty := math.Min(20, toF64(riskPositions)/math.Max(1, toF64(totalPositions))*100)
score := math.Max(0, 100-distancePenalty-stalePenalty-riskPenalty)
```

Agent 看不到这段代码。它只能调用工具，拿到一个数字，然后用自然语言"解释"一个它并不理解的东西。

这个问题不只是健康分。整个备件管理系统里有 20 多个指标，分散在 Go 代码、YAML 定义、设计文档和 Agent 的 system prompt 里，各说各话。

## 二、一个真实系统中的指标乱象

### 2.1 同一个目标，两个数字

备件管理有一个核心 KPI：总库存金额从 800 万降到 560 万，降幅 30%。这个数字来自现场调研，工段长的绩效考核直接挂在上面。

但代码里出现了两个不同的目标值：

| 位置 | 变量 | 值 |
|------|------|-----|
| `health.go` 第 37 行 | `targetValue` | 400,000（40 万） |
| `optimize.go` 第 75 行 | `targetValue` | 5,600,000（560 万） |

差了 14 倍。

这意味着：Agent 调用 `get_inventory_health` 时，健康分是按"目标 40 万"算的；调用 `get_optimization_plan` 时，优化方案是按"目标 560 万"算的。同一个对话里，Agent 可能先说"库存健康度还不错"，再说"距离目标还差 252 万"——自相矛盾，但它不知道。

如果"目标金额"这个参数有一个统一的、可查询的定义，这个问题就不会发生。

### 2.2 一个名字，三种口径

"呆滞"在备件管理中是一个关键概念——呆滞库存是降金额的主要来源之一。但"什么算呆滞"在系统中有三个不同的说法：

| 来源 | 定义 |
|------|------|
| 现场调研（调研人员 1） | 库龄 > 1 年 **且** 质保期过了 |
| 补充材料 | 只按时间计算，> 1 年 |
| Agent-4 的 system prompt | 呆滞阈值默认 **180 天**（demo 环境） |

更严重的是，`get_stale_items` 工具接受一个 `threshold_days` 参数，看起来可以自定义阈值——但代码里完全没有用这个参数。它直接查询 Neo4j 中 `isStale = true` 的节点，而 `isStale` 是种子数据手动设置的布尔值。

```go
// stale.go 第 21 行
records, err := d.Neo.QueryInventory(ctx, "", "", true)
// threshold_days 参数被接受但从未使用
```

这个参数给了 Agent 一个虚假的控制感。Agent 以为自己传了 `threshold_days: 365`，系统就会按 365 天过滤。实际上不管传什么，返回的都是同一批数据。

### 2.3 硬编码的业务判断

规则引擎中有三个阈值：

| 规则 | 阈值 | 来源 |
|------|------|------|
| R01 安全库存预警 | safety_stock = 2（所有备件统一） | 现场调研："所有备件统一备 2 个" |
| R04 高价值领用拦截 | unit_price > 2000 元 | 现场调研："> 2000 元备件损坏退库报废" |
| R05 频次异常检测 | 30 天内 ≥ 3 次 | 来源不明 |

这些数字直接写在 Go 代码里。当工段长问"为什么安全库存是 2"时，Agent 没有任何业务上下文可以引用。它不知道这个 2 是调研时确认的现场经验值，更不知道管理层的设计文档里已经提出了差异化方案——C 类备件降到 1、A 类备件根据采购周期动态计算。

### 2.4 设计了但不存在的指标

管理层决策支持的设计文档（v1.0）中定义了一个"库存结构四象限"模型：

```
口袋一：活库存 — 近 3 月有消耗 且 当前库存 ≤ 安全库存 × 2
口袋二：安全储备 — 关键性 A 且 (采购周期 > 30 天 或 停机损失极高)
口袋三：可优化过剩 — 当前库存 > 安全库存 × 2 且 不属于口袋二
口袋四：呆滞沉淀 — isStale = true
```

前端展示了这个四象限，数据是：活库存 312 万、安全储备 186 万、可优化过剩 228 万、呆滞沉淀 86 万。

这些数字全是硬编码在 `narratives.ts` 里的。没有任何后端工具计算这个分类。Agent 被要求在管理层看板中"透视库存结构"，但它调用的所有工具里都不存在这个计算。

Agent 会怎么做？它会编。它会根据有限的数据和自己的理解，凑出一个看起来合理的四象限分布。每次凑出来的数字可能都不一样。

还有更极端的例子。设计文档中出现了"断货概率 12%""75% 概率在 6 个月内达标""缺货风险 3.2%"这样的数字。这些数字在整个代码库中没有任何计算方法。它们是设计文档里的示例值，但 Agent 的 system prompt 引导它输出这类分析。结果就是 Agent 编造一个概率数字，管理者信以为真。

## 三、问题的本质

以上不是个别 bug，而是一个结构性问题：**本体定义了实体和关系，但没有定义衡量这些实体的标准。**

当前的 ontology YAML 中有四类元素：

| 元素 | 解决的问题 | 例子 |
|------|-----------|------|
| Classes | 世界里有什么东西 | 库存头寸、备件、库房 |
| Relationships | 这些东西之间什么关系 | 头寸位于库房、头寸跟踪备件 |
| Rules | 什么条件下触发什么动作 | 安全缺口 > 0 → 通知管家 |
| Actions | 可以对这些东西做什么 | 记录出库、生成请购建议 |

缺少的是第五类：**指标——用什么口径去衡量这些东西的状态。**

`derived` 属性部分填补了这个空白，但它只能描述单个实体上的简单计算。备件管理中真正困扰人的指标——健康分、四象限、呆滞率、达标进度——都不是单个实体的属性，而是跨实体的聚合分析，带有明确的业务口径、维度约束和参数依赖。

这类指标目前无处安放。它们散落在三个地方：

```
Go 代码（health.go, optimize.go, engine.go）
  → 有计算公式，但 Agent 看不到
  → 参数硬编码，无法解释业务含义

设计文档（管理层 AI 决策支持 v1.0）
  → 有业务定义，但没有对应实现
  → 示例值被当成了真实数据

Agent system prompt
  → 有模糊的指导，但没有精确口径
  → Agent 被要求输出它无法计算的东西
```

三个地方各说各话，没有一个地方是权威的。

## 四、为什么这个问题在 AI-Native 系统中尤为严重

在传统应用中，指标口径不统一的问题也存在，但通常不致命——因为指标的展示是确定性的。开发者写一段代码计算健康分，前端展示这个数字，用户接受就行。没有人会在运行时问"这个数字怎么来的"。

但在 Agent-Native 架构中，情况根本不同：

**Agent 会被追问。** 管理者不是在看一个静态报表，而是在和一个"参谋"对话。参谋说"健康度 72 分"，管理者一定会问"怎么算的""为什么不是 80 分""换一种算法会怎样"。Agent 必须能解释——但它只有一个数字，没有口径。

**Agent 会做推理。** Agent 不只是展示指标，它会基于指标做判断。"呆滞率 10.6%，建议先处置呆滞"——这个建议依赖"呆滞率"的口径。如果呆滞的定义从 365 天变成 180 天，呆滞率可能翻倍，建议就完全不同了。但 Agent 不知道它在用哪个口径做推理。

**Agent 会编造。** 当 Agent 被要求输出一个它无法通过工具获取的指标时，它不会说"我算不了"。大语言模型的本性是给出一个看起来合理的答案。"断货概率 12%"这个数字不来自任何计算，但它会被当成专业判断呈现给管理者。

**Agent 跨工具调用时缺乏一致性校验。** Agent 在同一个对话中调用 `get_inventory_health`（目标 40 万）和 `get_optimization_plan`（目标 560 万），它不知道这两个工具用了不同的假设。没有一个地方声明"目标金额"的权威值。

所以，指标口径的问题在传统应用中是一个数据治理问题，但在 Agent-Native 系统中是一个**信任问题**。管理者一旦发现 Agent 给的数字前后矛盾或无法解释，就会停止使用这个系统。

## 五、什么指标应该由本体定义

不是所有指标都应该放进 ontology YAML。判断标准是：**这个指标的口径是"业务领域的固有知识"还是"某个应用的产品设计"？**

### 5.1 属于本体的指标

这类指标描述的是备件管理这个领域本身的衡量标准。换一个应用、换一个客户，口径不应该变（参数可以变，但口径不变）。

**实体级派生指标**——对象本身的固有属性：

| 指标 | 口径 | 理由 |
|------|------|------|
| 可用数量 | currentQty - reservedQty | "可用量"在任何库存系统中的定义都一样 |
| 库存金额 | currentQty × unitPrice | 同上 |
| 安全缺口 | safetyStock - availableQty | "缺口"的定义是确定性的 |
| 库龄天数 | DATEDIFF(lastConsumedDate, NOW()) | 时间差计算，无歧义 |
| 月均消耗 | 近 N 月出库量 / N | 口径确定，参数 N 可配置 |
| 周转率 | monthlyAvgConsumption / currentQty | 库存管理的标准指标 |
| 库存支撑时长 | currentQty / monthlyAvgConsumption | 同上 |

**聚合分析指标**——跨实体的业务度量：

| 指标 | 口径 | 理由 |
|------|------|------|
| 呆滞率 | SUM(value WHERE isStale) / SUM(value) | "呆滞率怎么算"是业务定义 |
| 风险头寸数 | COUNT(positions WHERE safetyGap > 0) | "什么叫风险头寸"是业务定义 |
| 可释放金额 | SUM(currentQty - safetyStock) × unitPrice WHERE > 0 | "可释放"的口径是业务定义 |
| 库存四象限分类 | 见上文的四个口袋条件 | 这是备件管理领域的分析框架 |

**业务规则阈值**——判断标准的参数化定义：

| 阈值 | 默认值 | 理由 |
|------|--------|------|
| 呆滞天数 | 365 | "什么算呆滞"是领域知识 |
| 高价值门槛 | 2000 元 | 来自企业管理制度 |
| 异常频次 | 30 天 / 3 次 | 来自现场管理经验 |
| 月均消耗窗口 | 3 个月 | 行业常用统计窗口 |

### 5.2 不属于本体的指标

这类指标是特定应用的产品设计决策，换一个产品形态就不适用了。

| 指标 | 理由 |
|------|------|
| 健康分（100 - 三项扣分，权重 30/30/20） | 这个评分模型是产品团队设计的，权重分配没有业务依据。另一个备件管理产品完全可能用不同的评分模型 |
| 目标金额 560 万 | 这是永祥科技的经营目标，不是备件管理领域的通用概念 |
| 达标进度百分比 | 依赖客户特定的目标金额 |
| 月度金额趋势展示 | 把已有指标按时间排列，是展示层的选择 |
| Top N 消耗排名 | 排序和截断是产品交互设计 |

### 5.3 暂不定义的指标

| 指标 | 理由 |
|------|------|
| 断货概率 | 没有计算方法。放进本体也是空壳，反而让 Agent 以为自己有据可依 |
| 置信度评估 | 同上，需要统计模型支撑 |
| 决策回溯准确率 | 依赖 decision_log（beta 阶段），且评判标准未定义 |

这三个指标目前最正确的处理方式是：**在 Agent 的 system prompt 中明确说明这些指标暂无计算能力，不要编造。** 等有了方法论和数据支撑再加入本体。

## 六、本体的 derived 属性为什么不够

现有的 `derived` 字段可以描述实体级的简单公式：

```yaml
- id: safety_gap
  type: integer
  derived: "safety_stock - available_qty"
```

但它在以下场景中力不从心：

**缺少业务口径说明。** `derived` 只有公式，没有"这个指标是什么意思"的业务解释。当 Agent 被问"安全缺口是什么"时，它只能看到一个减法公式，不知道正值代表短缺、触发补货、影响安全评估。

**缺少参数化能力。** `monthly_avg_consumption` 的公式中写死了"3 个月"。这个 3 是业务参数还是不可变的口径？如果工段长想看 6 个月的均值呢？`derived` 没有区分"口径中可配置的参数"和"口径本身"。

**无法描述聚合指标。** "呆滞率"是所有头寸金额的加权比例，不属于任何单个实体。`derived` 绑定在 class 上，不能描述跨实体的聚合。虽然 YAML 有 `SUM([located_in].inventory_value)` 这样的跨关系聚合语法，但这只能做一跳聚合——四象限分类需要同时考虑消耗历史、关键性等级、采购周期，跨三个实体。

**无法描述分类逻辑。** 四象限不是一个数值计算，而是一个条件分类——每个头寸根据多个条件分到四个桶中。`derived` 的公式语法无法表达这种多分支逻辑。

**无法表达指标之间的依赖。** "可释放金额"依赖"安全库存"；"呆滞率"依赖"呆滞判定"的口径；四象限的"活库存"分类依赖"月均消耗"的计算窗口。这些依赖关系在当前 YAML 中是隐式的——如果改了月均消耗的窗口从 3 个月变成 6 个月，没有任何机制告诉你四象限的分类结果会受影响。

## 七、指标语义化需要什么

回到最初的问题：本体应该怎样承载指标？

一个指标在本体中的语义化定义，需要回答六个问题：

### 1. 这个指标叫什么、是什么意思？

```yaml
id: stale_ratio
name: 呆滞率
description: 呆滞库存金额占总库存金额的比例，反映库存资金的沉淀程度
```

这是给 Agent 看的。当管理者问"呆滞率是什么"时，Agent 可以直接引用这段定义。

### 2. 怎么算？

```yaml
formula: SUM(inventory_value WHERE is_stale = true) / SUM(inventory_value)
```

这是给 pipeline 看的。代码生成器将公式翻译成 SQL 或 Go 代码。

### 3. 算的时候有哪些可调参数？

```yaml
params:
  - id: stale_threshold_days
    name: 呆滞判定天数
    default: 365
    configurable: true
    description: 库龄超过此天数且无消耗的头寸被判定为呆滞
```

这解决了"三种口径"的问题。口径是确定的（按库龄天数判定），参数是可配置的（默认 365 天，客户可以改）。

### 4. 从什么维度去看？

```yaml
dimensions:
  - warehouse    # 可以按库房看呆滞率
  - category     # 可以按备件分类看呆滞率
granularity: warehouse  # 默认粒度
```

这告诉 Agent：当管理者问"哪个仓库呆滞最严重"时，这个指标支持按仓库维度切分。

### 5. 这个指标依赖什么？

```yaml
depends_on:
  - metric: is_stale_判定  # 呆滞率依赖"什么算呆滞"的定义
  - attribute: inventory_position.inventory_value
```

这让口径变更可追溯。如果"呆滞判定"的口径变了，系统知道"呆滞率"也会受影响。

### 6. 这个指标在当前实现中是什么状态？

```yaml
status: implemented          # implemented / designed / undefined
tool: get_stale_items        # 哪个 MCP 工具返回这个指标
known_issues:
  - threshold_days 参数目前未生效，查询直接使用 isStale 布尔值
```

这是给开发者和 Agent 共同看的。Agent 如果发现一个指标标记为 `undefined`，就应该明确告诉用户"这个指标暂无计算能力"，而不是编造一个数字。

## 八、回到备件管理：改变什么

如果上述六个维度的指标定义落地到备件管理本体中，以下问题会被解决：

**40 万 vs 560 万的矛盾消失。** "目标金额"被定义为应用层参数（不在本体中），所有引用它的地方从同一个配置读取，而不是各自硬编码。

**"什么算呆滞"有唯一答案。** 本体中声明口径是"库龄 > N 天无消耗"，参数 N 默认 365，可配置。Agent prompt、工具实现、前端展示都从这个定义生成。

**四象限不再是假数据。** 分类条件在本体中明确定义，pipeline 生成对应的计算工具，Agent 调用工具得到真实数据。

**Agent 知道自己能算什么、不能算什么。** 标记为 `undefined` 的指标（断货概率、置信度）不会出现在 Agent 的能力范围中。Agent 会说"这个指标目前无法计算"而不是编造一个数字。

**管理者追问时 Agent 有据可答。** "健康分怎么算的？"——即使健康分留在应用层，它引用的子指标（呆滞率、风险头寸比例）都有可查询的口径定义。Agent 可以逐层展开解释，而不是对着一个黑盒数字编故事。

## 九、Ontology 的定位：语义契约层

需要强调的是，这里讨论的指标语义化不是在建一个 BI 系统，也不是在扩展存储基础设施。

**Ontology YAML 是异构数据源和 Agent 之间的语义契约。** 它只定义"是什么"和"什么意思"，不管"怎么存"和"怎么算"。

无论是指标的计算引擎、物联网设备的时序数据存储、还是业务数据的关系型数据库——这些都有专门的服务或开源方案去负责。Ontology 本身只解决一个问题：**各种数据和 Agent 之间的识别沟通。**

这意味着 Ontology YAML 中不应出现：
- 存储选型（`graph_sync: true`、`graph_config`、`nodes_not_in_graph`）
- 实现方式（`implementation: sql/go/agent_delegated`）
- 数据源映射（`connector_hints`）
- 基础设施参数（归档天数、同步频率）

在这个定位下，分工变得清晰：

```
Ontology YAML（语义契约）：
├── 实体和关系的语义定义
├── 指标的口径、参数、维度、依赖关系
├── 规则和动作的语义描述
└── 指标的实现状态标记（implemented / designed / undefined）

Pipeline（语义胶水生成）：
├── MCP 工具接口定义（语义 → 工具契约）
├── Agent 能力绑定（语义 → Agent 配置）
├── 规则引擎配置（语义 → 执行规则）
├── 前端类型定义（语义 → TS 接口）
└── [可选] 存储 schema 插件（PG / Neo4j / TSDB）

各专业服务（能力实现）：
├── 关系型数据库 — 业务数据读写
├── 图数据库 — 图谱查询和可视化
├── 时序数据库 — 设备时序数据
├── IoT 平台 — 设备连接和数据采集
├── 计算引擎 — 指标计算和聚合
└── 通过 MCP 工具暴露能力，Agent 按语义契约调用

应用层：
├── 指标的展示格式（万元、百分比、小数位数）
├── 复合评分模型（如健康分的权重设计）
├── 客户特定的经营目标（如 560 万）
└── 本体与实例的图谱可视化展示
```

Agent 不需要知道呆滞率存在 PG 还是 Redis 里、温度数据存在 TDengine 还是 InfluxDB 里。它只需要知道：本体定义了一个叫"呆滞率"的指标，口径是什么、参数有哪些、可以通过哪个工具查询。

---

## 十、结论

这篇分析确认了两个结论：

**结论一：指标的语义化定义应该成为本体的一部分。** 它不是一个可选的增强，而是 Agent-Native 系统正确运行的前提。`metrics` 作为 YAML 的第三类语义元素（与 classes、relationships 并列），定义口径、参数、维度、依赖和实现状态。

**结论二：Ontology 是纯语义契约，不管存储和计算。** 指标的口径在本体中声明，指标的计算由专业服务实现，指标的展示由应用层决定。这和 classes 的处理方式一致——本体定义实体语义，存储由数据库服务负责，展示由前端负责。

两个设计原则：

**原则一：指标定义必须是 Agent 可查询的。** 不是写在注释里、不是写在文档里，而是通过 `query_ontology_metadata` 工具返回的结构化数据。Agent 能查到一个指标的口径、参数和状态，才能准确地回答用户的追问。

**原则二：不能定义的指标，就明确标记为不能定义。** "暂无计算方法"比"编一个数字"对管理者更有价值。指标的 `status: undefined` 标记不是系统的缺陷，而是诚实的声明。在 Agent-Native 系统中，可控的无知比不可控的幻觉安全得多。
