## 一、三个概念

本体设计只有三个核心概念。所有其他概念都从这三个概念派生。

### 本体（Ontology）

一个业务领域的完整语义描述。

"设备运维"是一个本体。"生产排程"是另一个本体。"合同管理"也是一个本体。它们各自描述了一个业务领域里有哪些东西、这些东西之间有什么关系、按什么规则运转。

类比：一个本体就像一张完整的地图。"杭州交通图"描述了杭州的交通世界——有哪些类型的地点、它们之间怎么连接、什么规则在起作用。同一座城市可以画不同的地图（交通图、地形图、行政区划图），同一个企业也可以有不同的本体，它们描述同一个现实但强调不同方面。

### 本体对象（Object Type）

本体里的一个业务对象类型。

在"设备运维"这个本体里，"设备""工单""备件""维修人员"各自是一个本体对象。它们不是四个本体，而是一个本体里的四个组成部分。在"合同管理"这个本体里，"合同""客户""付款节点""交付物"各自是一个本体对象。

类比：一个本体对象就像地图上的一种图例——"医院""学校""地铁站"。它定义了这片领地上有哪几类东西，以及每一类东西应该用什么符号和信息来描述。

### 本体实例（Instance）

本体对象的具体化，一条真实的业务数据。

"3号CNC机床"是"设备"这个本体对象的一个实例。"合同-HW20260326"是"合同"这个本体对象的一个实例。

类比：一个实例就像地图上的一个具体标注点——"浙一医院""凤起路站"。它是某种图例类型的一个真实存在。

### 三者的关系

```
本体（Ontology）
  │
  ├── 本体对象 A（Object Type）
  │     ├── 实例 a1
  │     ├── 实例 a2
  │     └── 实例 a3
  │
  ├── 本体对象 B（Object Type）
  │     ├── 实例 b1
  │     └── 实例 b2
  │
  └── 本体对象 C（Object Type）
        └── 实例 c1
```

各个元素的归属：

- **属性、规则、动作、函数**——归属于特定的本体对象。"主轴温度>85°C则停机"是设备这个本体对象的规则，"创建维修工单"是设备的动作。"付款逾期>30天触发预警"是合同的规则。
- **关系**——连接两个本体对象，不单独归属于哪一个。"设备→消耗→备件"这条关系属于本体整体。
- **接口**——跨本体对象的共享契约，属于本体整体。
- **安全策略**——可以定义在本体对象级别，也可以定义在本体级别。
- **指标**——归属明确的挂在本体对象上，跨对象的定义在本体级别。
- **遥测**——归属于产生数据的本体对象（如设备）。

> **Palantir 原文：**
> "An Ontology is a categorization of the world. In Foundry, the Ontology is the digital twin of an organization, a rich semantic layer that sits on top of the digital assets."
> ——[Palantir Foundry · Overview](https://www.palantir.com/docs/foundry/ontology/overview)
>
> "You can think of each object type as analogous to a dataset; an object is an instance of an object type, just as a row is one entry in a dataset."
> ——[Palantir Foundry · Core Concepts](https://www.palantir.com/docs/foundry/ontology/core-concepts)

> **学术界定义：**
> "An ontology is an explicit specification of a conceptualization."
> ——Gruber 1993 · [原始论文](https://tomgruber.org/writing/ontolingua-kaj-1993.pdf) · [2016修订版](https://tomgruber.org/writing/definition-of-ontology)

## 二、本体的构成元素

一个本体由以下元素构成。其中，类是核心——每个类通过属性、关系、规则、动作等元素来描述一类业务对象的完整语义。

### 核心元素

| 元素 | 说明 | 举例（以"设备"为例） |
|------|------|------|
| 类（Class） | 业务对象类型 | 设备、工单、合同 |
| 属性（Attribute） | 对象的特征描述，含静态属性、动态属性和计算属性 | 型号、运行工时、OEE |
| 关系（Relationship） | 对象间的语义联系 | 设备→位于→产线，设备→消耗→备件 |
| 规则（Rule） | 触发→条件→动作的业务规则链 | 振动 > 阈值 → 触发预警 |
| 动作（Action） | 受控的业务操作 | 创建维修工单、调整运行参数 |
| 函数（Function） | 复杂只读计算逻辑 | OEE = 可用率 × 性能率 × 良品率 |
| 接口（Interface） | 跨本体对象的共享契约和协作标准 | 响应排程查询、接受维修派单 |
| 安全策略（Security） | 多粒度权限控制 | 对象级、属性级、动作级权限 |
| 指标（Metric） | 跨对象的聚合分析视角 | 产线综合良品率、逾期合同占比 |
| 遥测（Telemetry） | 设备物联数据流的接入描述 | 主轴温度（1秒采样）、振动频率 |

前五个元素（类、属性、关系、规则、动作）是最基础的——有了它们就能描述"业务世界长什么样、怎么运转"。后五个元素（函数、接口、安全策略、指标、遥测）在特定场景下按需使用。

只有类、属性、关系的本体是一本词典。加上规则和动作的本体才是一个可执行的业务操作系统。

> **设计原则：** 规则只处理确定性判断（振动超阈值→预警、合同逾期→通知）。不确定性推理（故障根因是什么、该不该续约）交给 AI Agent。本体告诉 Agent "世界长什么样、什么规则在运转"，Agent 负责"在这个世界里做推理和决策"。

### 关于指标和遥测

**业务指标**分两种归属。归属明确的指标（设备的OEE、合同的履约率）是该对象的计算属性，放在属性里。跨对象的聚合指标（整条产线的综合良品率、本月逾期合同占比）不属于任何一个本体对象，而是本体层面的一个分析视角——就像地图上的"区域统计"（某区域内医院密度、平均通勤距离），描述的是一个区域的整体特征，不是某个标注点的属性。本体层面支持创建独立的指标类型的本体对象来承载这类跨对象指标。

**设备遥测**（温度、振动、电流等高频时序数据）概念上属于动态属性。但它的数据特征跟普通属性有本质区别——每秒一个值、有采样频率、有正常范围和告警阈值、查询时返回的是时间窗口的聚合值而不是单个值。在实现层面需要独立定义和处理，详见实现篇。

## 三、第一公民：设计的起点

设计一个本体时，建议先找到一个核心分析对象——我们称之为"第一公民"。它是管理者最关注的那个东西，是本体设计的自然起点。

第一公民不是本体的结构性规定，而是一个设计方法论工具：它帮你聚焦，让你知道从哪里开始往外扩展。在复杂场景下，不同角色的管理者可能关注不同的对象——维修主管关注设备，排程员关注工单，合同经理关注合同——这是正常的。选择第一公民是选择一个分析起点，不是宣布其他对象不重要。

举例：在设备运维场景中，第一公民可能是"设备"——维修主管盯的就是每台设备的健康状态、故障频次、下次保养时间。在生产排程场景中，第一公民可能是"工单"——排程员盯的是每个工单的进度、资源占用、交期风险。

识别第一公民后，建议关注它身上的三类信息：

| 信息类型 | 含义 | 举例（以"设备"为例） |
|---------|------|---------|
| 静态身份 | 它是谁 | 编号、型号、所属产线 |
| 动态状态 | 它现在怎么样 | 运行/停机/维修中、累计工时 |
| 行为画像 | 它的习性规律 | 月均故障次数、季节性负荷变化 |

## 四、怎么找到所有本体对象

选好设计起点之后，怎么找到它周围的其他本体对象？

方法：在建模之前，先用五个方向做一轮扫描。

制造业场景中，任何业务活动都可以归结为一句话：

> **某个资源（Resource），在某个空间/组织中（Topology），按照某个约定（Commitment），执行某个过程（Process），作用于某个对象（Object）。**

这五个方向是制造业的"基本粒子"——你的业务里一定有这五类东西。沿着它们去找，本体对象自然就出来了。

| 方向 | 问什么 | 举例 |
|------|--------|------|
| 资源 | 谁在做事？ | 设备、人员、工装夹具 |
| 对象 | 做事作用于什么？ | 物料、备件、能源介质 |
| 过程 | 做的是什么事？ | 加工工序、装配、搬运 |
| 约定 | 按什么约定做？ | 工单、生产计划、排班 |
| 拓扑 | 在哪里做？ | 车间、产线、工位、仓库 |

这不是建模本身，是建模前的"方向探测"。它告诉你"往哪几个方向去找"，然后你再对每个候选对象做深度分析——它的信号从哪里来、行动链路怎么闭环。

> **适用说明：** 五个扫描方向是制造业的通用框架。如果你的场景是合同管理、供应链金融等非制造领域，可以定义自己的扫描方向。核心要求是：互斥（一个对象只归入一个方向）、完备（能覆盖该领域绝大多数对象类型）。

## 五、设计方法：四个阶段

| 阶段 | 做什么 | 产出 |
|------|--------|------|
| ① 方向扫描 + 深度分析 | 选择核心分析对象作为起点，扫描五个方向定位候选对象，分析信号来源和行动链路 | 业务深度分析文档 |
| ② 现场验证 | 带着假设本体去调研：问故事、给选择题、标红线 | 已验证假设 + 确认的空白 |
| ③ 确定性梯度建模 | 确定性高的先建（核心对象和关系），不确定的标空白 | 最小可用本体（MVO） |
| ④ 迭代演进 | 用真实数据填充、用运行反馈修正、逐步扩展范围 | 生产级本体 |

核心节奏：先做确定性最高的那部分，但做到位。不追求一次覆盖全部业务。

## 六、五条设计原则

| 原则 | 一句话 |
|------|--------|
| 业务驱动 | 起点是"管理者脑子里在盯什么"，不是"系统里有什么表" |
| 动态优先 | 只有属性和关系的本体是半成品，必须加上规则和动作形成闭环 |
| 最小可用 | 先做 3-5 个核心本体对象，验证跑通后再扩展 |
| 确定性梯度 | 确定性的判断交给规则引擎，不确定性的推理交给 AI Agent |
| 语义与实现分离 | 本体定义只写业务语义，不写数据库怎么建、图谱怎么同步 |

## 七、本体与 AI Agent 的关系

Agent 需要本体来理解"业务世界长什么样"。没有本体的 Agent 只能聊天，有本体的 Agent 才能做事。

本体为 Agent 提供四样东西：

| 本体提供 | Agent 用来做什么 |
|---------|---------------|
| 统一的业务概念和语言 | 避免"同词不同义"——"工单"在所有 Agent 眼里是同一个意思 |
| 业务规则和约束 | 让推理"有据可依"——Agent 知道设备振动阈值是多少、合同逾期定义是什么 |
| 关系网络（因果链） | 沿着关系找到答案——从"设备故障"追溯到"哪些工单受影响" |
| 可执行的动作 | 从分析到行动——Agent 不只告诉你"该检修了"，还能创建维修工单 |

> **核心信念：** 建模的未来主体是大模型。人的角色是定义规范、审核结果、导入行业知识。大模型按照规范和知识库来生成结构化的本体，人用白盒方式打开、看懂、修改。
